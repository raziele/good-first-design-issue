# AI-First TDD Monorepo — Structural Guide

## Core Principle

**Specs are the source of truth. Tests are the contract. Code is the output.**

The human writes and maintains specs. Agent 1 generates tests from specs. Agent 2 writes code to pass those tests. Agent 3 reviews everything. The code agent *never* touches tests or specs.

## The Three Agents

### Agent 1 — Test Generator

**Input:** `specs/` directory (all chapters)
**Output:** `tests/` directory + `tests/manifest.json`
**Trigger:** Push to `specs/` on main branch

Responsibilities:
- Reads behavior specs → generates pytest tests (backend) and vitest/playwright tests (frontend)
- Reads API contracts → generates contract tests
- Reads UXI flows → generates e2e test skeletons and component tests
- Reads glossary/domain chapters → uses correct terminology in test names and assertions
- Maintains a **mapping manifest** (`tests/manifest.json`) linking each test back to its source spec section

Key design decisions:
- Tests should be **deterministic and hermetic** — no flaky network calls, use fixtures
- Each spec section should produce at least one test; the manifest tracks coverage
- Agent 1 should flag spec ambiguities as `TODO` comments in tests rather than guessing

### Agent 2 — Code Generator

**Input:** `tests/` directory (read-only) + `specs/` (read-only, for context)
**Output:** `src/` directory only
**Trigger:** After Agent 1 completes (or on baseline test failures)

Responsibilities:
- Runs the test suite, reads failures, writes/modifies code in `src/` to make tests pass
- Can read specs for context (understanding intent) but the *contract* is the tests
- Iterates: write code → run tests → fix → repeat (with a max iteration cap)

Constraints (enforced):
- **Cannot write to** `tests/`, `specs/`, `agents/`, `pipeline/`
- **Cannot delete tests** or modify test configuration
- **Cannot install new dependencies** without adding them to an allow-list
- **Max iterations per run:** configurable in `agents/agent-2-codegen/config.yaml`

### Agent 3 — Reviewer / Validator

**Input:** Git diff of Agent 2's changes + test results + specs
**Output:** Approve/reject report in `pipeline/reports/`
**Trigger:** After Agent 2 completes

Responsibilities:
- Verifies all tests pass (hard gate)
- Checks code quality: no dead code, follows project conventions
- Validates against brand chapter (naming, user-facing strings)
- Checks for spec compliance beyond what tests cover (e.g., "does the error message match the UXI tone?")
- Produces a structured report: pass/fail, issues found, confidence score
- Can **block merge** if confidence is below threshold (see `agents/agent-3-review/criteria.yaml`)

## Agent Harness: Cursor CLI

All three agents are invoked via the **Cursor `agent` CLI** — a stateless, timeout-bounded subprocess.

### How It Works

```
scripts/run_agent.py <agent-name>
  │
  ├── loads agents/<name>/config.yaml   → model, temperature, max_tokens
  ├── loads agents/<name>/prompt.md     → system prompt
  ├── reads pipeline/locks.yaml         → injects scope restrictions into prompt preamble
  └── calls: agent --api-key $CURSOR_API_KEY
                    --model <from config>
                    --output-format text
                    --trust --force
                    -p "<prompt>"
                    (cwd = project root, timeout = 900s)
```

### Model Configuration

Each agent's model is defined in its `config.yaml`. The default is `claude-sonnet-4-6`. To change a model, edit the relevant config file — no code changes needed.

| Agent | Config | Default Model |
|-------|--------|---------------|
| Agent 1 — Test Generator | `agents/agent-1-testgen/config.yaml` | `claude-sonnet-4-6` |
| Agent 2 — Code Generator | `agents/agent-2-codegen/config.yaml` | `claude-sonnet-4-6` |
| Agent 3 — Reviewer | `agents/agent-3-review/config.yaml` | `claude-sonnet-4-6` |

### Secret Management

The Cursor API key is stored locally under `.secrets/` (git-ignored) and uploaded to GitHub Actions via:

```bash
scripts/upload-secrets.sh
```

This requires `gh` CLI authenticated to the repo. It reads `.secrets/cursor-api-key` and sets `CURSOR_API_KEY` as a GitHub Actions secret.

## Filesystem Enforcement

### Permission Matrix

```yaml
# pipeline/locks.yaml
permissions:
  agent-1-testgen:
    read:  ["specs/**"]
    write: ["tests/**"]
    deny:  ["src/**", "agents/**", "pipeline/**"]

  agent-2-codegen:
    read:  ["specs/**", "tests/**"]
    write: ["src/**"]
    deny:  ["tests/**", "specs/**", "agents/**", "pipeline/**"]

  agent-3-review:
    read:  ["specs/**", "tests/**", "src/**"]
    write: ["pipeline/reports/**"]
    deny:  ["src/**", "tests/**", "specs/**"]
```

### Enforcement Layers

1. **Prompt preamble** — `scripts/run_agent.py` reads `locks.yaml` and injects scope rules at the top of every agent prompt: _"You may ONLY write to: X. You must NOT modify: Y."_
2. **CWD isolation** — Cursor runs with `cwd` set to the project root.
3. **Post-run validation** — `scripts/validate_scope.sh <agent-name>` runs after each agent invocation. It reads allowed write paths from `locks.yaml`, checks `git diff --name-only`, and fails CI if any out-of-scope file was modified.
4. **Git-based** — `CODEOWNERS` requires human approval for `specs/` and `tests/` changes.

## CI Pipeline

### Trigger

The pipeline runs on **push to `main` branch** when files under `specs/` change. Manual dispatch is also available (`workflow_dispatch`).

### Job Sequence

```
detect-changes
    │
    │  specs changed?
    ▼
agent-1-testgen  ──────────────────────┐
    │ (skipped if no spec changes)      │ artifact: generated-tests
    ▼                                   │
baseline-tests  ◄──────────────────────┘
    │  (runs even if agent-1 skipped)
    │  artifact: baseline-test-results
    ▼
agent-2-codegen
    │  artifact: codegen-src
    ▼
post-codegen-tests
    │  artifact: final-test-results
    ▼
agent-3-review
    │  artifact: review-report
    ▼
  (merge or flag for human review)
```

Each job:
1. Installs Cursor CLI: `curl https://cursor.com/install -fsS | bash`
2. Installs Python deps: `pip install pyyaml`
3. Invokes the agent: `python scripts/run_agent.py <agent-name>`
4. Validates file scope: `scripts/validate_scope.sh <agent-name>`

## Getting Started — Recommended Order

1. **Set up secrets** — add your Cursor API key to `.secrets/cursor-api-key`, run `scripts/upload-secrets.sh`
2. **Write `specs/spec-schema.md`** — define your conventions before writing specs
3. **Write the glossary and domain model** — these ground everything else
4. **Write 2–3 behavior specs** with Given/When/Then scenarios
5. **Test Agent 1 locally** — `CURSOR_API_KEY=... python scripts/run_agent.py agent-1-testgen`
6. **Validate scope enforcement** — `scripts/validate_scope.sh agent-1-testgen`
7. **Test Agent 2 locally** — `CURSOR_API_KEY=... python scripts/run_agent.py agent-2-codegen`
8. **Push to main** — CI triggers automatically on `specs/` changes
9. **Iterate on agent prompts** based on failure patterns

## Key Skills to Develop

- **Spec writing discipline:** Unambiguous, ID'd, cross-referenced. The specs are your codebase now.
- **Prompt engineering per agent:** Each agent has a carefully tuned system prompt in `agents/*/prompt.md`. Tune these based on observed failures.
- **Observability:** Every agent run produces a report in `pipeline/reports/`. Log every iteration.
- **Graceful degradation:** When Agent 2 can't make all tests pass within N iterations, it commits what it has, reports what's failing, and lets the human decide.
- **Spec versioning:** Tag spec versions so you can correlate "which spec produced which tests produced which code."
