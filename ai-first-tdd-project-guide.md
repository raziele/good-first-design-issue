# AI-First TDD Monorepo — Structural Guide

## Core Principle

**Specs are the source of truth. Tests are the contract. Code is the output.**

The human writes and maintains specs. Agent 1 generates tests from specs. Agent 2 writes code to pass those tests. Agent 3 reviews everything. The code agent *never* touches tests or specs.

## The Three Agents

### Agent 1 — Test Generator

**Input:** `specs/` directory (all chapters)
**Output:** `tests/` directory
**Trigger:** Spec file changes detected (via git diff on `specs/`)

Responsibilities:
- Reads behavior specs → generates pytest tests (backend) and vitest/playwright tests (frontend)
- Reads API contracts → generates contract tests
- Reads UXI flows → generates e2e test skeletons and component tests
- Reads glossary/domain chapters → uses correct terminology in test names and assertions
- Maintains a **mapping manifest** (`tests/manifest.json`) linking each test back to its source spec section

Key design decisions:
- Tests should be **deterministic and hermetic** — no flaky network calls, use fixtures
- Each spec section should produce at least one test; the manifest tracks coverage
- Agent 1 should flag spec ambiguities as `TODO` comments in tests rather than guessing

### Agent 2 — Code Generator

**Input:** `tests/` directory (read-only) + `specs/` (read-only, for context)
**Output:** `src/` directory only
**Trigger:** After Agent 1 completes, or on cron schedule

Responsibilities:
- Runs the test suite, reads failures, writes/modifies code in `src/` to make tests pass
- Can read specs for context (understanding intent) but the *contract* is the tests
- Iterates: write code → run tests → fix → repeat (with a max iteration cap)

Constraints (enforced):
- **Cannot write to** `tests/`, `specs/`, `agents/`, `pipeline/`
- **Cannot delete tests** or modify test configuration
- **Cannot install new dependencies** without adding them to an allow-list
- **Max iterations per run:** configurable (e.g., 10 cycles)

### Agent 3 — Reviewer / Validator

**Input:** Git diff of Agent 2's changes + test results + specs
**Output:** Approve/reject with comments, written to `pipeline/reports/`
**Trigger:** After Agent 2 completes

Responsibilities:
- Verifies all tests pass (hard gate)
- Checks code quality: no dead code, follows project conventions
- Validates against brand chapter (naming, user-facing strings)
- Checks for spec compliance beyond what tests cover (e.g., "does the error message match the UXI tone?")
- Produces a structured report: pass/fail, issues found, confidence score
- Can **block merge** if confidence is below threshold

## Filesystem Enforcement

### Hard Locks via CI

```yaml
# pipeline/locks.yaml
permissions:
  agent-1-testgen:
    read:  ["specs/**"]
    write: ["tests/**"]
    deny:  ["src/**", "agents/**", "pipeline/**"]

  agent-2-codegen:
    read:  ["specs/**", "tests/**"]
    write: ["src/**"]
    deny:  ["tests/**", "specs/**", "agents/**", "pipeline/**"]

  agent-3-review:
    read:  ["specs/**", "tests/**", "src/**"]
    write: ["pipeline/reports/**"]
    deny:  ["src/**", "tests/**", "specs/**"]
```

### Enforcement Mechanisms

1. **Git-based:** Use `CODEOWNERS` to require human approval for `specs/` and `tests/` changes. Agent 2's PRs are auto-scoped to `src/` only.
2. **CI-level:** The orchestrator runs each agent in an isolated environment (container or chroot) with mounted volumes matching the permission matrix above.
3. **Post-run validation:** A simple script diffs the worktree after each agent and rejects any out-of-scope file modifications before committing.

## Spec Authoring Conventions

### Recommended Spec Format

Use a consistent, parseable format so Agent 1 can reliably extract test cases:

```markdown
# Feature: User Authentication

## Context
- References: [glossary.md#session], [domain-model.md#user]
- UXI Flow: [flows/onboarding.flow.md]

## Rules

### RULE-AUTH-001: Password requirements
Passwords must be at least 12 characters with one uppercase,
one lowercase, and one number.

#### Scenario: Valid password
- Given: a registration form
- When: user enters "SecurePass123"
- Then: password is accepted

#### Scenario: Too short
- Given: a registration form
- When: user enters "Short1A"
- Then: error "Password must be at least 12 characters"
```

Key conventions:
- Every rule gets a **stable ID** (e.g., `RULE-AUTH-001`) so tests can reference it
- Scenarios use Given/When/Then for unambiguous test generation
- Cross-references link to glossary and UXI flows
- The `spec-schema.md` file documents these conventions as a contract

### UXI Flow Format

```markdown
# Flow: Onboarding

## Steps
1. User lands on /welcome
2. User fills name, email → clicks "Continue"
3. System sends verification email
4. User clicks link → redirected to /dashboard

## States
- Loading: skeleton placeholder for dashboard
- Error: inline message below email field
- Success: redirect with toast "Welcome, {name}!"

## Edge Cases
- Email already registered → show "Already have an account?" link
- Verification link expired → show re-send option
```

## Spec-to-Test Mapping (Manifest)

Agent 1 maintains `tests/manifest.json`:

```json
{
  "mappings": [
    {
      "spec": "specs/behavior/auth.spec.md#RULE-AUTH-001",
      "tests": [
        "tests/backend/unit/test_auth_password.py::test_valid_password",
        "tests/backend/unit/test_auth_password.py::test_too_short"
      ],
      "last_synced": "2026-03-28T10:00:00Z"
    }
  ],
  "unmapped_specs": [],
  "orphaned_tests": []
}
```

This manifest enables:
- **Coverage tracking:** Which specs have tests? Which don't?
- **Orphan detection:** Tests without a corresponding spec (spec was removed?)
- **Selective re-generation:** Only regenerate tests for changed specs

## Getting Started — Recommended Order

1. **Set up the monorepo skeleton** with the directory structure above
2. **Write `spec-schema.md`** — define your conventions before writing specs
3. **Write the glossary and domain model** first — these ground everything else
4. **Write 2–3 behavior specs** with Given/When/Then scenarios
5. **Build Agent 1's prompt** — start simple, generate tests for one spec, validate manually
6. **Set up filesystem locks** — even a simple bash script that checks git diff scope
7. **Build Agent 2's prompt** — give it the locked tests, let it write code
8. **Build Agent 3's prompt** — start with "all tests pass + no out-of-scope changes"
9. **Wire up the CI pipeline** — start with manual trigger, add cron later
10. **Iterate on agent prompts** based on failure patterns

## Key Skills to Develop

- **Spec writing discipline:** Unambiguous, ID'd, cross-referenced. The specs are your codebase now.
- **Prompt engineering per agent:** Each agent needs a carefully tuned system prompt with examples of good/bad output.
- **Observability:** Log every agent run, every test result, every iteration. You need to debug the *agents*, not just the code.
- **Graceful degradation:** When Agent 2 can't make all tests pass within N iterations, it should commit what it has, report what's failing, and let the human decide.
- **Spec versioning:** Tag spec versions so you can correlate "which spec produced which tests produced which code."
